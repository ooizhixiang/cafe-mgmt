## Intent

**Problem:** Each ingredient is currently tied to a single supplier with a single cost. Cafés source the same ingredient from multiple suppliers at different prices and units, and have no way to see or compare suppliers per ingredient or to review past purchases.

**Approach:** Replace `Ingredient.supplierId` with a many-to-many `IngredientSupplier` join carrying `priceInCents` and `unit` (default `"kg"`). Add an `IngredientPurchase` log. Surface both as a "Show all suppliers" disclosure on the expanded ingredient row in `/settings/ingredients` and `/inventory`. Manager has full CRUD; staff is read-only. When a supplier call is marked ORDERED, the call-outcome prompt offers an optional sub-form that creates an `IngredientPurchase` linked to that call.

## Boundaries & Constraints

**Always:**
- Server-side `requireRole("MANAGER")` on every write action; `requireAuth()` on read-only fetches; verify resources match `session.user.cafeId`.
- Money stored as integer cents; display as `RM X.XX` matching `formatRM` pattern.
- Keep `Ingredient.costPerUnitInCents` as the wastage/comp valuation cost — this field is independent of per-supplier prices and must NOT be removed or auto-derived.
- `(ingredientId, supplierId, cafeId)` is unique on `IngredientSupplier`. Same supplier can't be added twice to the same ingredient.
- Migration must backfill: for every existing `Ingredient` with non-null `supplierId`, create an `IngredientSupplier` row using its current `costPerUnitInCents` (or `0` if null) and `unit = "kg"` before dropping the `supplierId` column.

**Ask First:**
- If the migration backfill detects rows where `Ingredient.supplierId` references a soft-deleted or missing `Supplier`: HALT and report counts.
- If you discover existing call-log flows or seed scripts that depend on `Ingredient.supplierId`: HALT.

**Never:**
- Don't auto-update `IngredientSupplier.priceInCents` from purchase entries (manager controls list price; purchase prices are historical only).
- Don't change `dollar-attribution.ts` or any wastage/comp valuation logic.
- Don't add a "primary supplier" flag — the user explicitly rejected defaults.
- Don't expose CRUD UI to staff. Staff sees the suppliers + history tables read-only.
- Don't add `revalidatePath` calls; this codebase uses client-side state + `useTransition` + toast (match existing pattern).

## Design Notes

**Schema sketch (load-bearing):**
```prisma
model IngredientSupplier {
  id           String   @id @default(cuid())
  ingredientId String
  supplierId   String
  cafeId       String
  priceInCents Int
  unit         String   @default("kg")
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
  ingredient   Ingredient @relation(fields: [ingredientId], references: [id], onDelete: Cascade)
  supplier     Supplier   @relation(fields: [supplierId], references: [id], onDelete: Cascade)
  cafe         Cafe       @relation(fields: [cafeId], references: [id])
  purchases    IngredientPurchase[]
  @@unique([ingredientId, supplierId, cafeId])
  @@index([cafeId])
}

model IngredientPurchase {
  id                   String   @id @default(cuid())
  ingredientSupplierId String
  cafeId               String
  quantity             Int
  unit                 String
  totalPriceInCents    Int
  supplierCallLogId    String?
  createdById          String
  createdAt            DateTime @default(now())
  ingredientSupplier   IngredientSupplier @relation(fields: [ingredientSupplierId], references: [id], onDelete: Cascade)
  supplierCallLog      SupplierCallLog?   @relation(fields: [supplierCallLogId], references: [id], onDelete: SetNull)
  cafe                 Cafe     @relation(fields: [cafeId], references: [id])
  createdBy            User     @relation("PurchaseCreator", fields: [createdById], references: [id])
  @@index([cafeId, createdAt])
  @@index([ingredientSupplierId])
}
```
`SupplierCallLog` gets a back-relation `purchases IngredientPurchase[]`. `Cafe` and `User` get the obvious back-relations.

**Backfill SQL (in the migration):**
```sql
INSERT INTO "IngredientSupplier" (id, "ingredientId", "supplierId", "cafeId", "priceInCents", unit, "createdAt", "updatedAt")
SELECT gen_random_uuid()::text, i.id, i."supplierId", i."cafeId",
       COALESCE(i."costPerUnitInCents", 0), 'kg', NOW(), NOW()
FROM "Ingredient" i
WHERE i."supplierId" IS NOT NULL;
```
Then drop `Ingredient.supplierId`.

**Display ordering:**
- Suppliers table: ascending price (cheapest first).
- Purchase history: descending `createdAt` (newest first), include supplier name + qty + unit + total.
